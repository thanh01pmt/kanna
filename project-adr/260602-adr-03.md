### 1. Bối cảnh & Vấn đề cần giải quyết

Trong quá trình kiểm tra Workflow UI hiện tại, chúng ta phát hiện một điểm gây hiểu nhầm quan trọng:

*   Project mới hoàn toàn vẫn thấy workflow `Create Course` và các workflow đã publish trước đó.
*   Nút `Test Import` hiện chỉ tạo một manifest mock trong UI, chưa phải luồng import workflow thật.
*   `Start Workflow` đang trộn lẫn "workflow catalog" với "workflow sẵn sàng chạy trong project".

Điều này cho thấy kiến trúc hiện tại mới có:

*   **Workflow Catalog gần như global** (`workflow_definitions`, `workflow_versions`).
*   **Workflow Run theo project** (`workflow_runs`, `workflow_nodes`, `artifacts`, `artifact_impacts`).

Nhưng còn thiếu lớp trung gian quan trọng:

*   **Project Workflow Registry**: project này đã đăng ký/chọn những workflow nào, dùng version nào, workflow nào là default entrypoint.

### 2. Quyết định: Workflow Catalog, Project Registry, Workflow Run

Chúng ta thống nhất tách workflow thành 3 lớp:

1.  **Workflow Catalog**
    *   Nơi lưu các workflow đã import/publish.
    *   Workflow được import ở cấp Settings, không nằm trực tiếp trong project.
    *   Scope mặc định là theo user: user nào tạo workflow thì user đó dùng.
    *   Một số workflow có thể được chủ hệ thống publish global để mọi user thấy.
    *   Tương lai có thể cho user A publish workflow bằng workflow ID để user B import.

2.  **Project Workflow Registry**
    *   Project chỉ dùng các workflow đã được register vào project.
    *   User có thể tick chọn nhiều workflow từ catalog để thêm vào project.
    *   Mỗi project cần một workflow được chỉ định làm default entrypoint.
    *   Project registry pin vào một `workflow_version` cụ thể.
    *   Khi workflow global/catalog có version mới, project không tự upgrade. User phải bấm Upgrade, hoặc dùng setting rõ ràng nếu muốn auto-suggest/auto-upgrade.

3.  **Workflow Run**
    *   Một lần chạy cụ thể của một workflow version trong một project.
    *   Progress tree, artifacts, impact review, locks, và resume state đều thuộc project/run.

Scope đã chốt:

| Thành phần | Scope |
|---|---|
| Workflow definition/version | Theo user, hoặc global nếu official/global-published |
| Project workflow registration | Theo project |
| Registered version | Pinned version |
| Workflow run | Theo project |
| Artifact state | Theo project |
| Impact review | Theo project/run/artifact |

### 3. Workflow Authoring: Frontmatter là Contract, Extractor là Fallback

Workflow vẫn được viết bằng Markdown, nhưng phần YAML frontmatter sẽ là contract chính.

Frontmatter nên khai báo:

*   metadata: `name`, `version`, `description`
*   inputs
*   outputs/artifacts
*   dependencies
*   sub-workflows hoặc flow references
*   entrypoint/role
*   execution policy nếu cần

Markdown body là instruction vận hành cho agent.

Extractor vẫn tồn tại nhưng chỉ là fallback và assistant:

*   Nếu frontmatter đã khai báo artifacts, frontmatter là source of truth chính.
*   Extractor có thể bổ sung artifacts/dependencies dạng `inferred`.
*   Inferred artifacts cần user approve trước khi publish.
*   Nếu body nhắc artifact không khai báo trong frontmatter, UI import phải báo warning/error và cho user hiệu đính workflow definition.

Precedence:

```text
declared frontmatter > approved inferred metadata > raw heuristic extraction
```

### 4. Readiness Model: Artifacts/Files Are State

Chúng ta thống nhất cách tiếp cận:

```text
Project artifact state + workflow input contract + user instruction -> workflow can run
```

Một workflow chạy được khi input conditions thỏa mãn.

Input luôn là file state:

*   `file`: một file cụ thể.
*   `directory`: toàn bộ file trong directory đó.
*   `glob`: pattern file, ví dụ `units/U01/lessons/**/*.md`.

Readiness không chỉ dựa vào file tồn tại. Artifact input phải được hệ thống công nhận là:

```text
reviewed_ok / source_of_truth
```

Ngoài ra cần xét version/checksum:

*   Nếu input artifact mới hơn output artifact, output/downstream workflow phải vào trạng thái review/stale.
*   Nếu output đã có nhưng input đổi, workflow không tự rerun mù quáng. Hệ thống phải review downstream trước.

Workflow chỉ cần user prompt, không cần artifact input, là **initial-capable**, nhưng không tự động là project entrypoint. Cần khai báo rõ:

```yaml
entrypoint: true
role: initial
```

Utility workflow cũng có thể chỉ cần prompt nhưng không nên hiện trong project bootstrap nếu không phải entrypoint.

### 5. Flow Graph: Explicit Edges + Artifact IO Inference + AI Suggestions

Một project có thể có nhiều workflow: nối tiếp, song song, hoặc lồng nhau.

Graph nguồn có 3 lớp:

1.  **Explicit graph edges**
    *   Do workflow pack hoặc user khai báo trực tiếp.
    *   Ưu tiên cao nhất.

2.  **Artifact IO inferred edges**
    *   Hệ thống suy ra từ `inputs` và `outputs`.
    *   Ví dụ workflow B consume artifact do workflow A produce, vậy A -> B.

3.  **AI suggested edges**
    *   AI phân tích toàn bộ workflow trong project để đề xuất artifact nào được tạo trước/sau.
    *   AI suggestions cần user approve trước khi trở thành graph chính thức.

Precedence:

```text
explicit graph edges > artifact IO inferred edges > AI suggested edges
```

### 6. Nested Workflow UX

Do workflow lồng nhau có thể rất phức tạp, UI không nên ép user xem toàn bộ cây sâu trong một view.

Quyết định UX:

*   Tab tổng hiển thị main flow: `A -> B -> C`.
*   Nếu `A` có sub-workflow, click vào `A` sẽ mở detail tab/view của `A`: `A1 -> A2 -> A3`.
*   User thường chỉ cần biết flow tổng đang chạy node nào, node nào xong, và tạo ra artifact gì.
*   Detail view chỉ mở khi user muốn inspect hoặc review riêng một workflow/sub-workflow.

Hiện trạng implementation:

*   Schema đã có `nodes.children`, `produces`, `consumes`.
*   Runtime có thể render nested progress tree.
*   Nhưng chưa có orchestration engine thật sự để tự resolve/chạy workflow con.

### 7. Artifact Ownership, Output Conflict, Locks

Chúng ta thống nhất artifact canonical không nên bị nhiều workflow cùng tạo/sửa.

Quy tắc:

*   Mỗi canonical artifact có một owner workflow.
*   Workflow khác muốn sửa artifact đó phải đi qua review/repair request.
*   Shared/progress log artifact là exception, có thể append-only hoặc multi-writer.

Nếu có workflow A và M đều liên quan đến artifact X:

*   Nếu graph xác định A -> ... -> M, A luôn được ưu tiên chạy trước.
*   Nếu A và M ngang hàng rồi cùng feed vào X dạng shared/progress artifact, ai chạy trước cũng được nếu artifact được khai báo là shared/append-only.
*   Nếu hai workflow ngang hàng cùng muốn ghi canonical artifact X, đó là conflict cần resolve trước khi chạy.

MVP execution model:

*   Chạy cùng workspace.
*   Dùng lock theo artifact path/directory.
*   Nếu parallel thật sự hoặc sub-agent branch/worktree cần merge, triển khai sau.

Tương lai:

*   Sub-agent có thể chạy trên worktree riêng.
*   Kết quả merge về qua diff review.

### 8. Source of Truth, Review, Stale, Repair

Readiness dựa trên artifact đã được công nhận là source of truth.

Khi workflow tạo output artifact:

*   Mặc định output vào trạng thái `needs_review`.
*   User accept thì thành `source_of_truth`.
*   Nếu user cấp quyền cao cho AI, AI có thể tự quyết định artifact đủ quality để thành source of truth.

Cần có autonomy/permission model:

```text
manual_review
ai_recommend_user_approve
ai_auto_approve_low_risk
ai_auto_approve_all
```

Áp dụng được ở nhiều cấp:

*   global user setting
*   project override
*   workflow override
*   artifact criticality override

Khi input artifact đổi:

*   Downstream outputs/workflows chuyển `needs_review` hoặc `maybe_stale`.
*   AI diff đánh giá thay đổi của input artifact.
*   AI đề xuất downstream artifact nào `not_impacted`, artifact nào `needs_repair`.
*   User quyết định, trừ khi user đã cấp quyền cho AI tự approve/repair.

Không có "rerun toàn bộ" mù quáng. Luồng chuẩn là:

```text
input changed -> impact review -> AI recommendation -> user/AI approval -> repair if needed
```

### 9. Sequential vs Parallel Execution

User có thể quyết định:

*   chạy tuần tự các workflow ready
*   spawn sub-agent chạy riêng song song

Nhưng parallel cần policy:

*   workflow nào `parallel_safe`
*   artifact/directory nào đang bị lock
*   output artifact có conflict không
*   shared artifact có append-only/multi-writer không

MVP:

*   Không cần worktree song song phức tạp.
*   Dùng same workspace + lock.
*   Nếu file bị sửa bởi nhiều workflow, phải review/merge.

Tương lai:

*   Dùng worktree/branch riêng cho sub-agent.
*   Merge bằng diff review.

### 10. Resume Semantics

Hiện hệ thống có stateful tracking và partial resume semantics:

*   `workflow_runs`, `workflow_nodes`, `workflow_events`, `artifacts`, `artifact_impacts` lưu được state.
*   Mở lại project có thể reconstruct projection từ DB.
*   Có bước kiểu idempotent check để phát hiện artifact đã tồn tại và quyết định resume/repair/continue.

Nhưng chưa có workflow resume orchestration hoàn chỉnh:

*   Chưa có nút/flow `Resume`.
*   Chưa có scheduler xác định node nào blocked/ready/running sau reload.
*   Chưa có checkpoint input/output chính thức cho từng node/workflow.
*   Chưa có lock/retry/failover rõ ràng.
*   Chưa có resume sub-agent/job bị interrupted.

Quyết định tương lai:

*   Nếu run bị interrupted, UI hiện `Resume`, `Restart`, `Archive run`.
*   Resume phải re-evaluate file state, locks, readiness, và impact review trước khi tiếp tục.

### 11. Project UI cần tách rõ Catalog, Registered, Ready

UI hiện tại `Start Workflow` đang trộn nhiều khái niệm.

UI mới nên có:

1.  **Settings / Workflow Catalog**
    *   Import workflow Markdown.
    *   Review manifest.
    *   Publish version.
    *   View version history.
    *   Mark visibility: private user / official global / shared by ID.

2.  **Project Workflows**
    *   Tick chọn workflow từ catalog.
    *   Pin version.
    *   Set default entrypoint.
    *   Show update available nếu catalog có version mới.

3.  **Run/Start Panel**
    *   Chỉ hiện workflow đã register vào project.
    *   Phân loại:
        *   Ready now
        *   Blocked by missing/unreviewed inputs
        *   Running
        *   Needs review/stale
        *   Can repair/re-run

Project mới chỉ nên thấy:

*   suggested initial workflows từ registered/default entrypoint
*   hoặc màn nhắc register workflow nếu project chưa có workflow nào

### 12. Schema Gợi ý

Các bảng/field cần thêm hoặc làm rõ:

```sql
workflow_definitions (
  id,
  owner_user_id,
  slug,
  name,
  description,
  visibility, -- private_user | official_global | shared_by_id
  created_at,
  updated_at
)

workflow_versions (
  id,
  workflow_definition_id,
  version,
  status, -- draft | published | deprecated | archived
  source_markdown,
  manifest_jsonb,
  created_by,
  published_at
)

project_workflows (
  id,
  project_id,
  workflow_definition_id,
  workflow_version_id,
  enabled,
  is_default_entrypoint,
  alias,
  order_index,
  settings_jsonb,
  created_at,
  updated_at
)
```

Manifest cần mở rộng theo hướng:

```yaml
---
name: create-course
version: v3.1
description: Create a full curriculum course.
entrypoint: true
role: initial

inputs:
  prompt:
    required: true
  artifacts: []

outputs:
  artifacts:
    - id: curriculum_framework
      path: CURRICULUM_FRAMEWORK.md
      ownership: canonical

artifacts:
  - id: curriculum_framework
    name: Curriculum Framework
    pattern: CURRICULUM_FRAMEWORK.md
    owner: create-course

flow:
  edges:
    - from: create-course
      to: create-unit
      relationship: produces_input_for

execution:
  parallel_safe: false
---
```

### 13. MVP Quyết định Triển khai

Chúng ta thống nhất MVP không đi thẳng vào orchestration/parallel/resume phức tạp.

Thứ tự ưu tiên MVP:

1.  **Global/user workflow catalog trong Settings**
    *   Import workflow Markdown.
    *   Review manifest.
    *   Publish version.
    *   Version history.

2.  **Project workflow registration**
    *   Tick chọn workflow từ catalog.
    *   Pin workflow version.
    *   Chỉ project đã register workflow mới thấy workflow đó trong Start panel.

3.  **Default entrypoint**
    *   Mỗi project có một default entrypoint.
    *   Project mới chưa register workflow thì không tự hiện mọi workflow global.

4.  **Start panel đúng nghĩa**
    *   Chỉ hiện registered workflows.
    *   Hiển thị ready/blocked cơ bản.

5.  **Readiness đơn giản dựa trên source_of_truth**
    *   Required input artifact phải tồn tại và được accepted/source_of_truth.
    *   Output input-stale thì đưa vào review.

6.  **Sau MVP**
    *   Graph inference sâu bằng AI.
    *   Parallel/sub-agent worktree.
    *   Full resume orchestration.
    *   Workflow packs.
    *   Locks nâng cao.

### 14. Kết luận

Workflow platform của Kanna không nên là một danh sách global workflow hiện trong mọi project.

Kiến trúc đúng là:

```text
User/global catalog -> project registration with pinned version -> artifact-driven readiness -> run/projection/event store
```

Workflow là transition trên project state. Artifact/file là state. Readiness dựa trên artifact đã được công nhận là source of truth, không chỉ dựa vào file tồn tại.

ADR này chốt hướng MVP: trước tiên sửa tầng catalog/register/start để user hiểu đúng workflow nào thuộc project nào. Các phần orchestration nâng cao như graph inference, parallel sub-agent, locks nâng cao, và resume hoàn chỉnh sẽ đi sau.
