# ADR 05: Kanna as Agent Bridge, Permission Runtime, and Provider Capability Layer

## Status

Accepted for current implementation direction.

This ADR records the discussion and decisions around Kanna's role when working with multiple coding agents such as Claude, Codex, Antigravity (`agy`), Pi (`pi`), and future LLM-only backends.

## Context

The original idea was to use Pi Agent as the backend and Kanna as the UI. During implementation and testing, the design expanded: Kanna now needs to support multiple agent providers, normalize their behavior, and expose a consistent user experience.

The key questions raised were:

- Is Kanna only a proxy for agents, where the user can choose Claude, Codex, Antigravity, Pi, or another provider?
- How different are these agents in capability?
- How does Kanna reduce technical mismatch between providers?
- Should skills and MCP servers be attached to Kanna or directly to each agent?
- How should workflows work when not every agent has a native workflow concept?
- Can Kanna support a pure LLM backend?
- Is Kanna itself an agent with its own CLI?
- If Pi gets extra capability through extensions, can Kanna use it automatically?
- Could Kanna eventually replace external agents by giving Kanna CLI its own file/tool/runtime abilities?
- How can the user know whether an agent is actually running?
- How should Kanna handle CLI permission prompts from `agy` and `pi`?

## Decision Summary

Kanna is not just a thin UI over one backend agent. Kanna should become an orchestration/runtime layer with a consistent UI, transcript, tool approval model, workflow model, and provider capability abstraction.

External agents remain useful because they already contain mature model integrations, terminal/file tools, permission models, and coding-specific behavior. Kanna should use those capabilities through adapters instead of trying to immediately replace them.

However, Kanna should also keep a path toward becoming its own agent runtime:

- First: bridge external agents well.
- Then: normalize skills, MCP, permission, file access, and workflow semantics inside Kanna.
- Later: support LLM-only and Kanna-native CLI modes using the same runtime concepts.

## Provider Role

Kanna supports providers such as:

| Provider | Role in Kanna |
|---|---|
| Claude | Mature coding assistant provider; currently works in Kanna. |
| Codex | Coding agent provider with strong repository/file operation behavior. |
| Antigravity (`agy`) | Google Antigravity CLI provider; tested locally through CLI. |
| Pi (`pi`) | Pi Agent CLI provider; planned as a major backend candidate. |
| Pure LLM | Future backend mode where Kanna supplies tools/runtime instead of relying on an agent CLI. |

The correct abstraction is not "all providers are identical". The correct abstraction is:

```text
Provider-specific capability -> Kanna adapter -> Kanna normalized runtime/UI
```

## Capability Differences Between Agents

Agents differ across several dimensions:

- Model family and reasoning behavior.
- Native file editing ability.
- Native terminal/tool calling ability.
- Permission prompt model.
- Streaming format.
- Session/resume support.
- MCP support.
- Skill or plugin extension system.
- Workflow awareness.
- Output structure and reliability.

Kanna should not assume every provider supports the same features. Instead, each adapter should declare what it supports.

Example capability dimensions:

```ts
type ProviderCapabilities = {
  streaming: boolean;
  structuredEvents: boolean;
  fileTools: boolean;
  shellTools: boolean;
  permissionPrompts: boolean;
  mcp: boolean;
  skills: boolean;
  workflows: "native" | "kanna" | "none";
  resume: boolean;
};
```

This lets Kanna adapt behavior honestly instead of pretending all agents are equivalent.

## Reducing Provider Mismatch

Kanna "khử sai lệch" kỹ thuật by placing a normalization layer between the UI/runtime and each provider.

The normalization layer should handle:

- CLI command construction.
- Process lifecycle.
- stdout/stderr parsing.
- streaming text normalization.
- tool call normalization.
- permission request normalization.
- session and resume mapping.
- error and timeout reporting.
- provider-specific quirks.

The user should see one consistent Kanna transcript even if the underlying provider produces different event formats.

## Skills and MCP

Decision:

Skills and MCP should be declared and managed at the Kanna layer first, then exposed to providers when the provider supports them.

This means:

- Kanna should know the desired skills/MCP configuration for a project.
- If a provider supports native skills/MCP, Kanna adapter may pass the config through.
- If a provider does not support native skills/MCP, Kanna can still use those capabilities through Kanna-owned tools/runtime.
- Provider-specific extensions can still exist, but Kanna needs discovery or adapter updates to use them reliably.

In short:

```text
Declare in Kanna -> map to provider if supported -> fallback to Kanna runtime if not supported
```

## Configuration Ownership

Users should primarily interact with Kanna, not manually configure every provider for every project.

Decision:

Kanna should be the main control plane for project-level configuration:

- selected provider
- model/effort preference
- enabled skills
- MCP servers
- workspace directories
- workflow registry
- approval/autonomy settings

Provider-level configuration should only be required for provider-native prerequisites:

- login/authentication
- API key or account setup
- CLI installation/path
- provider-native plugin installation when Kanna cannot install/import it
- provider-specific global settings that cannot be represented by Kanna yet

Example:

If the user adds a skill in Kanna, Kanna should record it in Kanna project/runtime settings. Then:

- for a provider that supports native skills, the adapter maps it into the provider's native mechanism;
- for a provider that does not, Kanna uses the skill through its own runtime;
- if neither path exists yet, Kanna must clearly show the skill as unsupported for that provider.

This answers the practical Pi question:

```text
The user should not need to configure Pi manually for ordinary Kanna project behavior.
But Pi still needs its native installation/auth/plugin prerequisites.
Kanna is responsible for mapping Kanna settings into Pi when Pi exposes a supported interface.
```

## Provider Extensions

If Pi or Antigravity adds new functionality through extensions/plugins, Kanna should not need a hardcoded UI change for every small extension.

But Kanna can only automatically use extensions if the provider exposes machine-readable metadata, such as:

- available tools
- command schema
- permission needs
- input/output schema
- streaming event format
- plugin manifest

If the provider extension only changes the human CLI behavior, Kanna may still need adapter code changes.

Decision:

Kanna should prefer provider extension discovery through manifest/schema where possible. When unavailable, Kanna adapters must be updated explicitly.

## Workflows

Not every agent has a native workflow concept. Therefore workflow ownership belongs to Kanna, not to a specific agent.

Kanna workflow means:

- workflow catalog
- project workflow registry
- workflow runs
- artifact state
- source of truth review
- downstream review/repair
- event log
- projection UI

Agents are executors inside workflow steps. They may understand a workflow prompt, but Kanna owns the workflow state.

Decision:

```text
Workflow state lives in Kanna.
Agent providers execute tasks inside that workflow.
```

If a provider has native workflow support, Kanna may call it, but Kanna still records the canonical workflow state.

## Pure LLM Support

Kanna should support pure LLM mode in the future.

In pure LLM mode:

- The model only produces text/tool calls.
- Kanna owns file tools, shell tools, MCP tools, permission approval, and transcript.
- Kanna becomes closer to a real agent runtime.

This is a valid long-term direction, but it is not the same as simply calling an LLM API. To replace external agents, Kanna must implement or integrate:

- file read/write tools
- shell execution
- patch application
- permission model
- sandboxing
- streaming tool calls
- session memory
- project context packing
- workflow execution
- MCP/skills integration

Decision:

Kanna can support LLM-only backends, but removing all external agents requires Kanna CLI/runtime to become a full agent execution environment.

## Kanna CLI

Kanna should eventually have its own CLI, but the CLI should not be only a chat wrapper.

Kanna CLI should expose the same core runtime as the UI:

- start/resume chat
- select provider
- inspect provider status
- approve/deny pending permissions
- run workflow
- list artifacts
- inspect transcript
- run Kanna-native tools in pure LLM mode

Decision:

Kanna CLI is desirable as a control surface for Kanna runtime. It should share the same backend state and event model as the UI.

## Permission Bridge

The important correction from testing:

Kanna should not let `agy` or `pi` permission prompts block invisibly in a terminal or child process.

Correct bridge:

```text
agent CLI asks permission
-> Kanna parses permission request
-> Kanna emits normalized cli_permission_request tool call
-> UI shows approve/deny modal/message
-> user chooses
-> Kanna writes the selected choice back to the child process stdin
-> agent continues or stops
```

This is required for both `agy` and `pi`.

For Antigravity, observed prompt shape:

```text
Requesting permission for: openspec list

Do you want to proceed?
> 1. Yes
  2. Yes, and always allow in this conversation for commands that start with 'openspec'
  3. Yes, and always allow for commands that start with 'openspec' (Persist to settings.json)
  4. No
```

Kanna should normalize this into a `CliPermissionRequest` transcript/tool entry with provider, command, prompt text, and choices.

## Agent Running Status

The UI must make it clear whether a provider is actually running or blocked.

Important states:

- connected to Kanna server
- session started
- provider process spawned
- provider streaming output
- waiting for permission
- waiting for user input
- timed out
- errored
- interrupted
- completed

The earlier issue was that the UI showed `Running...` for several minutes while Antigravity was blocked or launched incorrectly.

Decision:

Kanna must expose provider process state separately from general chat connection state.

Minimum useful UX:

- `Running...` while process is active and output is flowing or expected.
- `Waiting for permission...` when a parsed CLI permission request is pending.
- `No output yet` or timeout warning if process produces no output for a threshold.
- provider error message when the child process exits non-zero.

## Antigravity CLI Findings

The local `agy --help` output showed that the original adapter assumption was wrong.

Observed `agy` command shape:

```text
agy --print "hello"
```

works.

Important help output:

```text
--print                         Run a single prompt non-interactively and print the response
--print-timeout                 Timeout for print mode wait (default 5m0s)
--prompt-interactive            Run an initial prompt interactively and continue the session
--continue                      Continue the most recent conversation
--conversation                  Resume a previous conversation by ID
--dangerously-skip-permissions  Auto-approve all tool permission requests without prompting
```

The earlier implementation tried:

```text
agy run --model gemini-3.5-flash --output-format jsonl --non-interactive ...
```

This is not supported by the observed local CLI.

Then the adapter tried `agy --print` but passed the prompt through stdin. The local CLI expects the prompt as a command argument, so the process could hang.

Current decision:

Use local-help-aligned CLI invocation:

```text
agy --print --print-timeout 5m "<prompt>"
```

For resume:

```text
agy --print --print-timeout 5m --conversation <conversation-id> "<prompt>"
```

The model/effort controls in Kanna's UI should not be blindly passed to `agy` unless the CLI or SDK exposes supported flags. Until then, those controls are either decorative for Antigravity or must be mapped through a supported configuration mechanism.

## Antigravity SDK Direction

Antigravity provides an official SDK repository:

- `google-antigravity/antigravity-sdk-python`

Longer term, the SDK may be a better bridge than scraping CLI text because it may provide:

- structured events
- tool calls
- permission hooks
- session handling
- model/config control

Decision:

Keep the CLI adapter as the immediate working path. Investigate SDK adapter later for a more native and reliable integration.

## Pi CLI Direction

Pi should follow the same Kanna bridge principles:

- Do not assume Pi uses the same JSONL shape as Codex.
- Build the adapter around the actual local CLI contract.
- Parse permission prompts if Pi emits interactive approval requests.
- Normalize Pi output into Kanna transcript entries.
- Map Pi extensions through manifests/schema if Pi exposes them.

Current adapter direction:

```text
pi --mode json --print --no-session --model <model> [--thinking <effort>] [--session-id <id>] "<prompt>"
```

This must be verified against the installed Pi CLI in the local environment. If Pi changes CLI behavior, the adapter should be updated to match observed help/docs.

## Duplicate Message Issue

Observed issue:

Typing `xin chào` and sending could create duplicate user messages.

Decision:

The chat input should guard against double submit at the UI level.

Implementation direction:

- Add an in-flight submit ref.
- Ignore additional submit attempts while a send is already in progress.
- Reset the guard when the send completes or fails.

This prevents accidental double-send from Enter/click race conditions or repeated event handlers.

## Current Implementation Notes

Changes implemented during this discussion include:

- Added normalized `cli_permission_request` tool kind.
- Added shared transcript/tool hydration support for CLI permission requests.
- Added server-side pending tool support for CLI permission responses.
- Wired Antigravity and Pi managers to receive `onToolRequest`.
- Added client UI component for CLI permission approval/deny.
- Routed UI approval back through `chat.respondTool`.
- Added chat input double-submit guard.
- Refactored Antigravity adapter to spawn per turn using `agy --print --print-timeout 5m "<prompt>"`.
- Refactored Pi adapter to spawn per turn using observed/expected print mode shape.
- Adjusted Antigravity output handling so raw text can still become assistant output when structured JSON is unavailable.

Verification performed:

- `pnpm --filter @kanna/server check` passed.
- `pnpm --filter @kanna/client build` passed.

Known caveat:

- The running dev server must be restarted after server adapter changes. Old child processes such as stale `agy run` or old `agy --print` sessions can leave the UI stuck in `Running...`.

Suggested restart commands:

```text
pkill -f "agy --print"
pkill -f "agy run"
bun run dev
```

## Final Architecture Direction

Kanna should be treated as:

```text
Kanna UI
  -> Kanna runtime/coordinator
  -> Provider capability adapters
  -> External agents or pure LLM runtime
```

Provider adapters are not an implementation detail only. They are part of Kanna's core architecture because they define how Kanna can safely and honestly use each agent.

The stable long-term model:

```text
Kanna owns:
- UI
- transcript
- workflow state
- artifact state
- permission approval
- MCP/skills declaration
- provider capability registry
- Kanna CLI/runtime

Providers own:
- their native model behavior
- their native tools
- their native plugin system
- their native session model

Adapters translate:
- provider-specific process/events/prompts
- into Kanna-normalized events, tool calls, status, and permissions
```

## Open Questions

- Should Antigravity use CLI bridge only, SDK bridge only, or both with fallback?
- How should Kanna discover provider plugins/extensions in a machine-readable way?
- Which provider capabilities should be user-visible in Settings?
- Should Kanna show unsupported provider controls as disabled instead of decorative?
- What is the minimum Kanna-native tool runtime needed before pure LLM mode is useful?
- Should permission approvals be persisted per project, per provider conversation, or global user settings?
- How much provider process telemetry should be shown in the main chat versus a debug panel?

## Consequences

Positive:

- Kanna avoids being locked to Pi, Claude, Codex, or Antigravity.
- Users get one consistent UI and workflow model.
- Permission prompts become visible and auditable.
- Provider quirks are isolated in adapters.
- Kanna has a credible path toward becoming a full agent runtime.

Tradeoffs:

- Adapters require maintenance as provider CLIs change.
- Some provider features will not be automatically available unless exposed by schema/manifest.
- Pure LLM support is more work than calling an API because Kanna must provide the missing agent runtime.
- Workflow semantics must remain Kanna-owned even when providers have their own planning or workflow behavior.
