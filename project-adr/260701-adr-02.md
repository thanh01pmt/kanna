### 1. Kiến trúc Dữ liệu & Event Store (Data Architecture)

Chúng ta đã quyết định chuyển từ SQLite (chỉ dùng làm local adapter) sang **Postgres (Supabase)** làm nền tảng chính để hỗ trợ multi-user và versioning. Kiến trúc tuân thủ nghiêm ngặt mô hình **CQRS / Event Sourcing**:

**A. Nguồn sự thật (Source of Truth - Event Store):**
Dữ liệu được ghi dưới dạng *append-only* (chỉ thêm mới, không ghi đè) để đảm bảo khả năng audit và debug.
*   **Bảng `workflow_events`**: Lưu mọi sự kiện xảy ra.
    *   *Các loại event đã chốt:* `workflow_started`, `step_discovered`, `step_started`, `artifact_produced`, `artifact_changed`, `impact_review_started`, `repair_finished`, v.v.
    *   *Schema:* `id`, `run_id`, `sequence`, `type`, `payload_jsonb`, `created_at`, `actor_type` (user/agent/system), `actor_id`.

**B. Trạng thái hiện tại (Projections - View cho UI):**
UI không query trực tiếp bảng events mà đọc từ các bảng Projections được DB tự động build:
*   **Bảng `workflow_nodes`**: `id`, `run_id`, `parent_id`, `node_type` (workflow/task/step/gate), `status`, `source` (imported/discovered/dynamic), `children_sealed`.
*   **Bảng `artifacts`**: `id`, `project_id`, `path`, `kind`, `version`, `checksum`, `produced_by_node_id`.
*   **Bảng `artifact_impacts`**: `source_artifact_id`, `impacted_artifact_id`, `status`, `reason`.

### 2. Thiết kế UI Right Sidebar & Quản lý Trạng thái (UI/UX & State)

**A. Hiển thị số lượng Step (Trung thực với dữ liệu):**
*   **Tuyệt đối không hiển thị tổng số step ảo** (VD: `12/42 steps`) khi workflow chưa được khám phá hết (chưa *sealed*).
*   **Từ vựng UI (Vocabulary) đã chốt:**
    *   `known` / `materialized`: Đã biết chắc chắn sẽ chạy.
    *   `horizon open`: Dự kiến sắp tới, chưa materialize.
    *   `conditional` / `dynamic`: Sinh ra dựa trên điều kiện.
*   *Định dạng hiển thị chuẩn:* `12 done · 1 running · 2 known next · horizon open`. (Chỉ dùng `steps` khi `childrenSealed === true`).

**B. 3 Chế độ hiển thị (Density Modes) của Sidebar:**
1.  **Compact:** Chỉ hiện Live step + Progress summary.
2.  **Normal (Mặc định):** Live step + Current phase tree + Artifacts + Attention/Gates.
3.  **Expanded/Detail:** Click vào step/artifact để xem log chi tiết, dependencies, và các nút action (Rerun/Repair).

### 3. Quản lý Artifact & Cơ chế Downstream Repair

Đây là điểm khác biệt cốt lõi so với các hệ thống chạy AI thông thường. Không có khái niệm "Rerun toàn bộ", mà phải phân tách rõ 6 hành động (Semantics):

1.  **Rerun:** Chạy lại đúng step tạo ra artifact đó.
2.  **Review downstream:** Giữ artifact đã sửa làm *source of truth*, kiểm tra các artifact phụ thuộc xem có bị lệch không.
3.  **Repair downstream:** Sửa artifact phụ thuộc bị lệch (patch nhỏ hoặc regenerate).
4.  **Regenerate:** Tạo lại hoàn toàn.
5.  **Invalidate:** Đánh dấu là lỗi thời.
6.  **Accept as source of truth:** Chấp nhận thay đổi thủ công của user.

**Luồng xử lý khi Artifact bị sửa:**
*   *Sửa nhỏ (Typo):* Auto review downstream -> Đánh dấu `reviewed_ok`.
*   *Sửa lớn (Cấu trúc):* Đánh dấu các downstream là `needs_repair` -> **Hỏi ý kiến user** trước khi chạy repair hàng loạt.
*   *Schema ImpactReview:* `changedArtifactId`, `impactedArtifactId`, `relationship` (direct/transitive), `status` (needs_review / reviewed_ok / needs_repair / repaired / obsolete).

### 4. Workflow Manifest & Import Pipeline

Không bắt user cấu hình tay Artifact Kind. Workflow khi import phải trải qua Pipeline 2 tầng để sinh ra Manifest:

1.  **Static Extractor:** Script parse file Markdown để bóc tách các file output rõ ràng (VD: `LESSON_{unit}_{lesson}.md`).
2.  **AI Workflow Analyzer:** AI đọc toàn bộ Markdown để suy luận ngữ nghĩa (Semantic Role) và sự phụ thuộc (Downstream Impact).
3.  **Human Approval:** Con người duyệt bản Draft Manifest -> Publish thành **Immutable Workflow Version**.

**Schema Workflow Manifest:**
*   `ArtifactDefinition`: `kind`, `pathPattern`, `producedBy`, `consumedBy`, `semanticRole`, `required`.
*   `ArtifactDependencyRule`: `sourceKind`, `targetKind`, `relationship` (defines/informs/validates/generates), `impact` (must_review / maybe_review / ignore_minor).

### 5. Tầm nhìn Trung hạn: Artifact Intelligence & Reuse Marketplace

Biến hệ thống thành kho tri thức có thể định giá và tái sử dụng.

**A. Phân loại & Tìm kiếm (Stack: Postgres + pgvector):**
*   Mỗi artifact sinh ra sẽ được AI Worker chạy ngầm: Classify (phân loại), Extract concepts (rút trích khái niệm), Compute embeddings (tạo vector ngữ nghĩa), Detect duplicates (tìm trùng lặp).
*   **Bảng `artifact_relationships` (Graph layer ban đầu):** `source_artifact_id`, `target_artifact_id`, `relationship_type` (assesses / belongs_to / covers / similar_to / derived_from), `confidence`.

**B. Luồng Tái sử dụng (Reuse Flow) khi có Order mới:**
1.  Normalize request -> Compute fingerprint.
2.  **Exact match:** Tìm request/checksum giống hệt.
3.  **Semantic search:** Tìm artifact có ngữ nghĩa tương đương (qua pgvector).
4.  **Graph filter:** Lọc theo grade/topic/difficulty.
5.  **UI Offer:** Đề xuất user: `[Use existing - cheaper]` (Dùng lại giá rẻ), `[Adapt existing]` (Điều chỉnh nhẹ), `[Generate new]` (Chạy AI tạo mới).

**C. Quyền sở hữu (Reuse Policy):**
*   Cần các trường: `reuse_scope` (private / org / marketplace / global) và `reuse_mode` (exact / adapted / reference_only / prohibited).

### 6. Tầm nhìn Dài hạn: Personalized Learning Platform

Chuyển trọng tâm từ Workflow sang **Learner Model** (Người học) và **Recommendation Engine** (Động cơ đề xuất).

**A. Learning Record Store (LRS) Schema:**
*   `concepts`, `learning_objectives`, `content_items` (lesson/question/quiz).
*   `learning_events`: Lưu hành vi (mở bài, xem hint, nộp bài).
*   `answer_attempts`: Lưu chi tiết câu trả lời, thời gian, độ chính xác.
*   `learner_mastery`: Đánh giá năng lực (`mastery_score`, `confidence`, `next_review_at`).

**B. Recommendation Engine (5 Tầng):**
Khi user trả lời sai, hệ thống đề xuất nội dung học bù (remediation) qua 5 bộ lọc:
1.  **Rule-based:** Đúng cấp độ, ngôn ngữ, quyền truy cập.
2.  **Mastery-based:** Dựa trên lỗ hổng kiến thức (concept yếu) và Spaced Repetition.
3.  **Graph-based:** Tìm bài học nằm trên đường dẫn tiên quyết (prerequisite path).
4.  **Vector-based:** Tìm nội dung có ngữ nghĩa tương tự với lỗi sai hiện tại.
5.  **AI Reranker:** AI giải thích lý do đề xuất và sắp xếp lại top candidates.

### 7. Những gì đã thực sự Code & Triển khai (Achievements)

1.  **Package `@kanna/workflow-tracker`:** Đã code xong UI Right Sidebar với các token design chuẩn, logic đếm step cục bộ (`childrenSealed`).
2.  **Tích hợp Supabase:** Đã tạo `.env.local`, chạy migration thành công tạo các bảng cốt lõi. Viết xong `SupabaseWorkflowRuntimeStore` tự động fallback về in-memory nếu mất kết nối.
3.  **Package `@kanna/mcp` (Model Context Protocol):** Đã code xong MCP server stdio với 7 tools chuẩn:
    *   `workflow_list_definitions`, `workflow_start_run`, `workflow_get_projection`, `workflow_list_runs`, `workflow_list_events`, `workflow_append_event`, `artifact_list`.
    *   *Ý nghĩa:* Agent giờ đây tương tác với workflow qua API chuẩn xác, không cần đoán mò UI.
4.  **Dev Monorepo Ergonomics:** Đã sửa script `dev` chạy 2 port rõ ràng: `[kanna] app: http://localhost:5174` (Vite UI) và `[kanna] backend: http://localhost:5175` (Bun Server).
5.  **Đồng bộ hóa Real-time (Real-time Synchronization):** Đã thiết lập tính năng lắng nghe thay đổi DB thời gian thực (Supabase Realtime) cho các bảng liên quan đến workflow (runs, nodes, artifacts, impacts) và tự động đồng bộ hóa đăng ký (just-in-time subscription) thông qua `ws-router.ts` trên Bun backend, giúp UI Workflow Tracker Panel tự động cập nhật ngay khi trạng thái thay đổi.
