# ADR 06: Antigravity CLI Integration Fixes & Streaming Normalization

## Context
Kanna integrates with the Antigravity (`agy`) CLI via a Node.js child process (`child_process.spawn`). The integration was previously unstable, exhibiting symptoms where the UI would remain stuck in a "Running..." state indefinitely, and text output from the model would either not stream or appear to hang silently.

During a deep debugging session, two fundamental structural issues were identified in how Kanna interacted with the `agy --print` command over standard I/O pipes.

### Issue 1: The Piped Context Deadlock (`stdin` blocking)
When `agy` is spawned programmatically by `child_process.spawn` with `stdio: ["pipe", "pipe", "pipe"]`, the Go-based CLI detects that `os.Stdin` is a pipe. In its default behavior, it attempts to read the entirety of this piped stream to append to the prompt context (e.g., `cat data.txt | agy --print "summarize"`).
- Kanna held the `stdin` stream open indefinitely so that it could theoretically respond to permission prompts in the future.
- Because `stdin` was never closed via `.end()`, `agy` blocked indefinitely at `io.ReadAll(os.Stdin)`, creating a deadlock before any model generation could begin.

### Issue 2: Line-Buffered Streaming (`stdout` text dripping)
When in `--print` mode, `agy` utilizes a "text drip" mechanism (`text_drip.go`) to simulate typing in the terminal. This mechanism outputs individual characters or words without appending newline (`\n`) characters until the very end of a paragraph or response.
- Kanna's streaming parser (`handleRawChunkBuffer`) was splitting incoming chunks strictly by newlines (`/\r?\n|\r/g`) to parse JSON records.
- Incomplete lines were appended to a `trailing` variable. Because the dripping text lacked newlines, the entire response was cached silently in Kanna's backend buffer (`trailing`) and was never flushed to the frontend UI transcript until the stream completely closed (which it couldn't, due to Issue 1).

## Decisions

### 1. Close `stdin` Immediately for CLI Mode
To prevent the Piped Context Deadlock, Kanna must close the `stdin` pipe immediately after spawning the `agy` process in CLI mode:
```typescript
if (process.env.KANNA_ANTIGRAVITY_TRANSPORT !== "sdk") {
  session.child.stdin.end();
}
```
**Consequence:** Because `stdin` is closed, Kanna can no longer answer interactive CLI permission prompts (e.g., `[y/N]`). The CLI mode must now rely on `--dangerously-skip-permissions` to bypass prompts, or risk failing/panicking if a tool requires approval. 

### 2. Implement Real-Time Chunk Flushing for Non-JSON Output
To handle "text drip" behavior in CLI mode, the stdout buffering logic was modified to detect when incoming streams are not emitting JSON lines. If a chunk cannot be parsed as JSON and originates from `stdout`, the buffer (`trailing` text) is flushed directly to the UI as an `assistant_text` segment in real-time, completely bypassing the newline requirement.

### 3. Shift Towards the Python SDK Bridge as the Primary Contract
Due to the brittle nature of screen-scraping a TTY-oriented Go CLI—specifically regarding the handling of permission prompts through standard I/O—the long-term architectural decision is to treat the **Python SDK Bridge** (`KANNA_ANTIGRAVITY_TRANSPORT=sdk`) as the robust, production-ready path. The SDK outputs standard JSON-lines and handles tool orchestration properly, eliminating stream buffering and IPC deadlock issues.

## Status
Accepted and Implemented (as of `2026-06-02`).

## Consequences
- The Antigravity CLI transport in Kanna now streams text in real-time and completes gracefully without hanging.
- Interactive CLI permission requests via Kanna UI are fundamentally incompatible with the CLI bridge due to the `stdin` EOF requirement.
- Future robust tool-use for Antigravity will require finalizing the SDK bridge (`antigravity-sdk-bridge.py`).
